Final First

13 Jun 2014

Tags

  • #programming

Share

Programming languages are mostly introduced according to a common scheme (with minor author or language specific variations). You may check this by buying several books for different programming languages like Java, C#, C++, C, Ruby, Python and so on. After you have studied these books you will identify a very similar table of contents in first chapters across all these books. To shorten this process I advise you may want to read the book "Seven languages in Seven Weeks" from Bruce Tate and you will find the following aspects covered for every programming language.

  • Read from and write to console (the famous "Hello World")
  • Variables and data types
  • In modern (script) languages often collections (lists and mappings, depending on the programming language more complex concepts like classes are necessary, than this is normally introduced later but typically in first semester)
  • Operators and expressions
  • Control structures (sometimes including exception handling if the language provides a exception handling but most langugages provide exception handling nowadays)
  • Defining routines (methods, procedures, functions including maybe recursive programming or even anonymous functions or closures, if you have teachers that are open for a functional programming style)
  • I/O programming (to provide essential file and maybe networking operations)
  • Object oriented constructs (introducing classes and objects because you need these concepts for a lot of more sophisticated concepts in following semesters)

If this looks familiar to you, it is likely that you understand at least one programming language and you got the core concepts almost every popular programming language provides nowadays.

According to my knowledge this is exactly what is done at most computer science study programmes around the world in programming courses for freshman. And not very astonishing I do exactly the same in my Programming I course. Feel free to check this by checking my course materials.

So you may ask if this is done at most universities around the world what is wrong with this approach?

Whats wrong with it

From a Programming I course point of view which should have the primary intend to convey very fundamental concepts of programming everything is fine. The problems arise later. Take for example my agenda for Programming II which is a second semester course.

My programming II course covers more complex and sophisticated aspects (and I admit we have much more thematic variations here across computer science study programmes from my experience):

  • Intensified object oriented programming (especially subtyping polymorphism and Liskovian substitution principle and maybe including object oriented analysis and design)
  • Testing the correctness of programs (or to be more precise proofing the existence of bugs)
  • Generic datatypes (which can be skipped if we use a language capable of parametric polymorphism like Haskell - nevertheless we have to deal with teaching parametric polymorphism as well, which also can be tricky)
  • Graphical user interfaces
  • Concurrent (multithread) programming

And here the core problem can be seen. Complex concepts like concurrent programming are introduced later than fundamental and more easier understandable concepts. This is a core concept of teaching.

But most programming courses violate the concept to start with something simple, manageable and to introduce more sophisticated, harder understandable and manageable concepts later on at their very first examples of programming.

If you are familiar with programming these lines of Java code should be obvious to you. If you are not familiar with programming languages the meaning of the following lines are explained.

Scanner in = new Scanner(System.in);
    System.out.println(
      "Hello, please insert your name: "
    );
    String name = in.nextLine();
    String greet = "Good morning";
    greet += " " + name;
    System.out.println(greet);

The program simply asks a user to input its name (via keyboard) and answers with "Hello Tom" (if the entered name was "Tom") on a text based console.

It is a very simple variation of "Hello World" used to enable freshman to write code which is capable to provide (simple) interactions with a user.

So what is wrong with the code above? On a first look nothing! It provides exactly the intended behaviour. User is asked to enter a name (e.g. "Tom") and is greeted by mentioning his or her entered name (e.g. "Good morning Tom").

The problem is that we have introduced (without real need) a very complex concept of mutables. We introduce three variables

  • in to provide an object to handle keyboard interactions of the user
  • name to store the entered name of the user
  • greet to generate a welcome text for the user (e.g. "Good morning Tom")

And we do this to structure the code into more understandable pieces simply to be able to explain the code line by line. And it is great because we can introduce concepts that programs are executed operation by operation (which in these simple introduction examples is nearly line by line). We can also introduce data types and so on using a very simple and understandable example.

But to do this we could also have written this code here to do exactly the same.

final Scanner in = new Scanner(System.in);
System.out.println("Hello, please insert your name: ");
final String name = in.nextLine();
final String greet = "Good morning";
System.out.println(greet + " " + name);

We only introduced one additional keyword final. And due to the meaning of the Java keyword final we now introduced three immutables (immutables are something like variables which can be assigned only once a value) instead of variables. In fact final can often been seen as something like an abbreviation for complex expressions (so a non changing value).

In fact we could have even written the code in the following form

System.out.println("Hello, please insert your name: ");
System.out.println("Good morning " + (new Scanner(System.in)).nextLine());

which makes it more obvious (for experienced programmers) that there are no changeable variables any more.

But normally we are too slothfull to explain one additional meaning of the keyword final or to explain complex expressions (at this early point of a programming course). In fact we are happy at this early point of a lecture to have introduced console based interactions.

We introduce mutables without thinking about the consequences

But in fact we teach students from their very first days of programming to use a concept (mutable state) which is proofed to be harmful (especially in concurrent programming situations). And so students get used to this style of programming without really reflecting it (because at this point we do not have explained how tricky this style of programming will become in more complex scenarios of concurrent programming).

From my experience students often grip to this style of programming to their end of their studies. Which is sad. They often deny the possibility to solve problems with changing values by only using immutables.

And this can be seen already several weeks after introducing the concept of variables. Another very famous programming excercise is "Guess a number". Students are asked to implement a program showing the following behaviour. The program determines a random number between 1 and 100. And the user is asked to guess the number. If the users guess is smaller than the random number the program answers "My number is bigger." and vice versa. The user is asked repetitively to guess the number until he or she has guessed the randomly determined number.

Due to the fact that we have changing values here (the number to guess) students use variables.

Normally implement this exercise is implemented like that:

Random r = new Random();
Scanner in = new Scanner(System.in);

int secret = r.nextInt(100) + 1;

int nr;

do {
  System.out.println("Please guess a number between 1 and 100: ");
  nr = in.nextInt();
  if (nr == secret) {
    System.out.println("Congratulations. This was my number. ")
  }
  if (nr < secret) {
    System.out.println("My number is bigger.");
  } else {
    System.out.println("My number is smaller.");
  }
} while (nr != secret);

I have never seen a solution like that (which would indicate that a student is really aware of mutables and immutables).

final Random r = new Random();
final Scanner in = new Scanner(System.in);

final int secret = r.nextInt(100) + 1;

int nr; // this is the only mutable
      // we need if we want to solve this by using a loop

do {
  System.out.println("Please guess a number between 1 and 100: ");
  nr = in.nextInt();
  if (nr == secret) {
    System.out.println("Congratulations. This was my number. ")
  } else if (nr < secret) {
    System.out.println("My number is bigger.");
  } else {
    System.out.println("My number is smaller.");
  }
} while (nr != secret);

By using methods we could even skip the last mutable nr.

/**
 * Method to capsule the guessing logic.
 * @param secret - the number to guess
 * @param in - the scanner object to read the keyboard
 */
public static void guess(final int secret, final Scanner in) {
  System.out.println("Please guess a number between 1 and 100: ");
  final nr = in.nextInt();
  if (nr == secret) {
    System.out.println("Congratulations. This was my number. ")
  } else if (nr < secret ) {
    System.out.println("My number is bigger.");
    guess(secret, in); // Guess one more time.
  } else {
    System.out.println("My number is smaller.");
    guess(secret, in); // Guess one more time.
  }
}

public static void main(String[] args) {
  final Random r = new Random();
  final Scanner in = new Scanner(System.in);
  final int secret = r.nextInt(100) + 1;

  guess(secret, in);
}

But we do not train students to avoid mutables. In fact

  • mutables are not seen as code smells,
  • mutables are not seen as last chance solutions.
  • Mutables are seen as something normal.

Instead of this common point of view mutables should be seen as very special features of programming languages which should be used rarely and with a lot of care.

In fact we should teach students to use final first and train students to avoid mutables where ever possible.
